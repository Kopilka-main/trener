## Приоритетные «быстрые победы»

Сначала закрыть простые FE-only задачи, которые видимо меняют UX и не требуют расширения API:

1. **A1+A2** — порядок и переименование плиток (15 мин)
2. **D8** — типы контактов (1 коммит, иконки + переключатель)
3. **E1** — пресеты интервалов в бухгалтерии (30 мин)
4. **B1+B2** — relax-тема + переключатель (1-2 коммита)
5. **A3** — баннер уведомлений на главной (30 мин)

## Средние задачи

6. **A5+A4** — обновлённая шапка профиля и блок финансов на главной
7. **C1+C2** — фильтры и подразделы базы знаний (+ поле equipment в БД)
8. **D5** — список проведённых тренировок в карточке клиента
9. **D6** — автоматический архив пакета

## Крупные блоки

10. **D1+D2+D3** — статистика клиента с графиками (подходы/тоннаж/объёмы)
11. **D7** — медкарта с файлами
12. **D4** — фото клиента до/после
13. **F1** — swipe-back

## Замечания по архитектуре

- **Файловое хранилище** (D7, D4) — текущий бэкенд на SQLite + Express без файлового стораджа. Варианты:
  - Локальный диск + раздавать через `express.static('/uploads')` (просто, но не для production).
  - Внешний сервис: S3, Cloudinary, Supabase Storage.
- **Графики** (D1, D2, D3) — нужна либа: `recharts` (React-friendly, ~50KB gzipped) или нативный SVG через `d3-scale`.
- **Календарь измерений** (D3) — новая таблица `client_measurements` (date, height, weight, body_fat%, обхваты).
- **Equipment field** (C1) — новое поле в `exercises` + миграция БД.
- **Архивный статус пакета** (D6) — серверная логика: пакет переходит в `status='archived'` когда `paid <= completed`. Можно сделать триггером БД или вычислять на лету.
